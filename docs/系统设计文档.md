## 4. 关键业务流程

### 4.1 任务状态流转

```
                    ┌──────────┐
                    │  待开始   │ (todo)
                    └────┬─────┘
                         │ 开始执行
                         ▼
                    ┌──────────┐
           ┌─────── │  进行中   │ (in_progress) ────────┐
           │        └──────────┘                       │
           │ 完成编码          │ 提交测试               │ 发现问题
           ▼                   ▼                        ▼
    ┌──────────┐       ┌──────────┐             ┌──────────┐
    │ 已延期    │       │  待测试   │             │  已关闭   │
    │(delayed) │       │(to_test)  │             │ (closed)  │
    └──────────┘       └────┬─────┘             └──────────┘
                            │ 测试通过
                            ▼
                       ┌──────────┐
                       │  已完成   │
                       │  (done)   │
                       └──────────┘
```

状态转换规则：
- todo -> in_progress：开始执行任务
- in_progress -> to_test：提交测试
- in_progress -> delayed：确认延期
- in_progress -> closed：任务取消/关闭
- to_test -> in_progress：测试不通过，返工
- to_test -> done：测试通过
- done -> closed：任务归档关闭
- delayed -> in_progress：继续执行

### 4.2 Bug 状态流转

```
                    ┌──────────┐
                    │   新建    │ (new)
                    └────┬─────┘
                         │ 确认
                         ▼
                    ┌──────────┐
           ┌─────── │  已确认   │ (confirmed) ──────┐
           │        └────┬─────┘                    │
           │ 分配修复     │                          │ 确认为非问题
           ▼             ▼                          ▼
    ┌──────────┐  ┌──────────┐              ┌──────────┐
    │ 暂不处理  │  │  修复中   │              │ 暂不处理  │
    │ (wontfix) │  │ (fixing)  │              │ (wontfix) │
    └──────────┘  └────┬─────┘              └──────────┘
                        │ 修复完成
                        ▼
                   ┌──────────┐
                   │  已修复   │
                   │  (fixed)  │
                   └────┬─────┘
                        │ 验证关闭
                        ▼
                   ┌──────────┐
                   │  已关闭   │
                   │ (closed)  │
                   └──────────┘
```

状态转换规则（代码中严格校验）：
- new -> confirmed：确认问题存在
- new -> wontfix：确认为非问题或暂不处理
- confirmed -> fixing：开始修复
- confirmed -> wontfix：暂不处理
- fixing -> fixed：修复完成
- fixing -> wontfix：放弃修复
- fixed -> closed：验证通过，关闭

## 5. 安全设计

### 5.1 RBAC 权限模型

系统基于角色的访问控制（Role-Based Access Control）分为三层：

1. **角色定义**：通过 `roles` 表定义，预置 admin、teacher、student 三个角色。
2. **用户关联**：每个 `user` 通过 `role_id` 外键关联一个角色。
3. **权限控制**：通过 `@role_required` 装饰器在路由层面控制访问。

```python
def role_required(*roles):
    """限制路由访问角色"""
    def decorator(f):
        @wraps(f)
        def decorated_function(*args, **kwargs):
            if not current_user.is_authenticated:
                return abort(401)
            if current_user.role.name not in roles:
                return abort(403)
            return f(*args, **kwargs)
        return decorated_function
    return decorator
```

### 5.2 密码安全

- 使用 bcrypt 算法进行密码哈希存储，非明文存储。
- 登录错误 5 次后自动锁定账号（`is_active = False`）。
- 密码修改需要验证原密码。

### 5.3 Session 安全

- 使用 Flask-Login 管理用户 Session。
- 生产环境配置 `SESSION_COOKIE_HTTPONLY = True`。
- CSRF 保护默认启用（WTF_CSRF_ENABLED = True）。

### 5.4 其他安全措施

- 文件上传限制 16MB（`MAX_CONTENT_LENGTH`）。
- Nginx 反向代理隐藏应用服务器信息。
- Gunicorn 生产模式部署，非 debug 模式。
- 用户输入经过 WTForms 和服务端双重校验。
