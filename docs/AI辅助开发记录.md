## 3. 使用场景与案例

### 3.1 场景一：数据库表结构设计

**使用工具**：通义灵码（Tongyi Lingma）

**提示词**：
> "帮我为云智实训管理平台生成 MySQL 建表 SQL，需要包含用户表、角色表、课程表、项目表、小组表、任务表、日报表、Bug表、爬虫配置表、AI使用记录表、评分表等，使用 InnoDB 引擎，utf8mb4 字符集。"

**AI 输出摘要**：生成了完整的 DDL 脚本，包含 16 张表的结构定义、主外键关系、索引设计、ENUM 枚举类型定义。

**处理方式**：直接采用 AI 生成的表结构框架，人工调整了以下内容：
- 修改了部分字段类型（如将 `VARCHAR(255)` 改为 `TEXT` 以支持长文本）
- 增加了 `password_hash` 字段以支持 bcrypt 加密存储
- 补充了联合唯一约束（如 `uk_team_user`、`uk_user_date`）
- 增加了 SHA256 数据去重哈希字段

**效果评价**：AI 生成的 DDL 提供了非常好的基础框架，节省了约 60 分钟的手写时间。但需要人工审核字段类型选择和约束条件。

### 3.2 场景二：Flask 用户登录路由

**使用工具**：豆包（Doubao）

**提示词**：
> "帮我用 Flask 写一个用户登录路由，使用 Flask-Login，密码用 bcrypt 验证，登录失败 5 次后锁定账号。需要记录登录日志。"

**AI 输出摘要**：生成了基本的路由框架，包含登录表单验证、密码校验、LoginLog 记录逻辑。

**处理方式**：采用了 AI 生成的基本结构，但进行了以下修改：
- 重写了密码验证逻辑（将 `check_password_hash` 改为自定义的 `check_password` 方法）
- 增加了登录失败计数和账号锁定机制
- 优化了用户提示信息（显示剩余尝试次数）
- 增加了 `next` 参数安全校验（防止开放重定向漏洞）

**效果评价**：AI 提供了 60% 的可用代码，安全相关逻辑（开放重定向防护、SQL 注入防护）需要人工补充。

### 3.3 场景三：CSS 页面样式优化

**使用工具**：通义灵码

**提示词**：
> "优化任务看板页面的 CSS 布局，使用 CSS Grid 实现 6 列看板布局，支持响应式，卡片样式要简洁现代。"

**AI 输出摘要**：生成了基于 Grid 的看板布局 CSS，包含卡片阴影、过渡动画、响应式断点。

**处理方式**：采用 AI 生成的 Grid 布局方案，手动调整了间距和颜色方案以符合 Bootstrap 5 的主题风格。卡片样式直接可用。

**效果评价**：CSS 样式生成效果很好，节省了大量调试时间。AI 对现代 CSS 布局（Grid/Flexbox）的掌握程度很高。

### 3.4 场景四：SQLAlchemy 错误解释

**使用工具**：ChatGPT

**提示词**：
> "Flask 应用报错 sqlalchemy.exc.IntegrityError: (pymysql.err.IntegrityError) (1452, 'Cannot add or update a child row: a foreign key constraint fails')，这是什么原因？"

**AI 输出摘要**：解释了外键约束冲突的三种常见原因：引用的主键不存在、级联操作顺序错误、NULL 值冲突。给出了调试建议。

**处理方式**：直接采用 AI 的解释来定位问题。实际原因是测试数据插入顺序不对——先插入了引用外键的数据，被引用的数据还没插入。

**效果评价**：AI 对数据库错误的解释非常精准，问题定位时间从 15 分钟缩短到 2 分钟。

### 3.5 场景五：测试用例生成

**使用工具**：豆包

**提示词**：
> "生成 Flask 任务管理模块的测试用例，需要覆盖任务 CRUD、状态流转、权限控制、边界条件。"

**AI 输出摘要**：生成了 20 条 pytest 测试用例，覆盖正常流程和异常边界。

**处理方式**：采用了 AI 生成的测试用例框架和大部分场景，补充了以下测试：
- 非法状态转换测试（如 "todo" 直接跳到 "closed"）
- 角色权限验证测试
- 并发场景测试

**效果评价**：AI 生成的测试用例覆盖了主要功能路径，节省了大量构思测试场景的时间。

### 3.6 场景六：Dockerfile 编写

**提示词**：
> "为 Flask 项目写一个 Dockerfile，使用 python:3.11-slim 基础镜像，需要优化构建缓存层。"

**AI 输出摘要**：生成了分层的 Dockerfile，将 requirements.txt 安装放在代码复制之前以充分利用 Docker 缓存。

**处理方式**：直接采用 AI 生成的 Dockerfile 方案，仅调整了工作目录路径。这个生成结果非常完美。

### 3.7 场景七：docker-compose.yml 编写

**使用工具**：豆包

**提示词**：
> "为 Flask + MySQL + Redis + Nginx 项目写 docker-compose.yml，包含 healthcheck、volume 挂载、network 配置。"

**AI 输出摘要**：生成了四服务编排的 compose 文件，包含 MySQL healthcheck、数据卷持久化、自定义网络。

**处理方式**：采用了 AI 生成的基本框架，进行了以下调整：
- 增加了 MySQL 初始化脚本挂载（自动执行 schema.sql）
- 增加了服务依赖关系（web depends_on db condition: service_healthy）
- 增加了端口映射调整（MySQL 映射到 3307 避免主机冲突）

**效果评价**：docker-compose 的结构和基本配置可用，但健康检查和服务依赖的细节需要人工修正。

### 3.9 场景九：nginx 错误日志分析

**使用工具**：ChatGPT

**提示词**：
> "Nginx 日志报错 'connect() failed (111: Connection refused) while connecting to upstream'，客户端看到 502 Bad Gateway，是什么原因？"

**AI 输出摘要**：指出了三种可能原因：Flask 应用未启动、端口不匹配、Docker 网络配置错误。

**处理方式**：直接采用 AI 的分析结果，检查后发现是 Docker 容器启动顺序问题——Nginx 在 Flask 应用就绪前就开始请求。通过增加 `depends_on` 和健康检查解决了问题。

**效果评价**：日志分析效率大幅提升，AI 能快速定位常见的运维问题。

### 4.3 使用建议

1. **AI 擅长结构性工作**：生成代码框架、DDL、测试用例、文档大纲等结构化文本，AI 表现最好。
2. **安全逻辑必须人工审查**：认证、授权、输入校验等安全相关代码，AI 输出存在盲区，必须人工复审。
3. **错误调试效率提升最明显**：AI 解释报错信息的准确率很高，能将问题定位时间缩短 80% 以上。
4. **长期项目需记录 AI 使用痕迹**：系统内置的 ai_records 模块帮助学生记录 AI 使用过程，便于教师评估 AI 在实训中的作用。
5. **提示词质量决定输出质量**：好的提示词应包含上下文、具体需求、输出格式要求，能显著提升 AI 输出质量。
