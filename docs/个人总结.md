## 4. 遇到的挑战与解决方案

### 4.1 挑战一：数据库外键约束设计

**问题**：设计 16 张表的外键关系时，发现表之间存在复杂的循环引用和级联删除冲突。例如 Bug 表要先于任务表创建（因为任务表引用 Bug），但 Bug 表又引用了用户表。

**解决方案**：按照"基础表 -> 业务表 -> 关联表"的依赖顺序逐层建表。明确了外键删除策略的选择：核心业务数据用 SET NULL，日志类数据用 CASCADE。调整了建表顺序，确保被引用的表先创建。

### 4.2 挑战二：任务看板状态流转设计

**问题**：任务状态有 6 种（todo/in_progress/to_test/done/delayed/closed），Bug 状态有 6 种（new/confirmed/fixing/fixed/closed/wontfix），状态之间的转换关系复杂，需要防止非法转换。

**解决方案**：为每个模块设计了显式的状态转移表（valid_transitions dict），在路由层做严格校验。非法转换时返回错误提示而不是静默失败。这个设计在测试阶段发现并阻止了多处非法状态变更。

### 4.3 挑战三：Docker 容器间通信

**问题**：首次部署时，Flask 容器无法连接到 MySQL 容器，报错 `Can't connect to MySQL server on 'db'`。

**解决方案**：排查后发现两个问题：一是 MySQL 容器启动需要时间，Flask 启动时 MySQL 还没就绪；二是需要在 docker-compose.yml 中设置 `depends_on` 和 `condition: service_healthy`。最终通过健康检查机制解决了启动顺序问题。

### 4.4 挑战四：权限控制的细粒度

**问题**：同样是 "查看日报" 这个操作，学生只能看自己的，教师可以看全班，管理员可以看所有。同样的模式在 AI 记录、Bug 等模块中也存在。

**解决方案**：在路由层通过 `current_user.role.name` 判断角色，动态拼接查询条件，而不是写多个独立的视图函数。通过装饰器实现粗粒度控制，在视图函数内部实现细粒度数据过滤。

## 5. 个人反思

### 5.1 做得好的

1. **前期规划充分**：数据库设计和蓝图结构花了较多时间，后续开发非常顺畅，几乎没有结构性调整。
2. **自动化优先**：从项目一开始就配置了 Docker 环境，避免了 "在我机器上能跑" 的问题。
3. **代码即文档**：Python 代码有完整的类型注解，路由函数的命名和注释清晰，减少了单独的接口文档需求。
4. **充分利用 AI**：通过通义灵码、豆包、ChatGPT 等工具辅助开发，整体效率提升约 40%。

### 5.2 可以改进的

1. **测试滞后**：功能开发完后才集中测试，导致有些问题发现较晚。后续应先写测试再写功能。
2. **前端能力不足**：前端页面虽然功能完整，但在交互体验（拖拽、动画、实时更新）方面还有提升空间。
3. **提交粒度不够细**：Git 提交日志不够规范，有时一次提交包含了多个功能的改动，不利于代码审查。
4. **安全知识需要加强**：虽然实现了基本的 CSRF 防护和密码加密，但在 XSS 防护、SQL 注入防护方面还需要更系统地学习。

### 5.3 对实训过程的思考

通过这次实训项目，我深刻体会到：

- **需求理解是基础**：花在理解需求上的时间永远值得，方向对了后面就不会走弯路。
- **AI 是工具不是替代**：AI 能帮我们写代码、查错误、生文档，但架构设计、安全策略、业务逻辑的关键决策还是需要人来把控。
- **自动化是 DevOps 的核心**：Docker 部署、日志管理、备份脚本这些自动化工作，虽然前期投入时间，但长期来看极大地提升了开发和运维效率。
- **文档是最好的总结**：写文档的过程就是对项目的全面回顾。通过写这些文档，我发现了项目中一些可以优化的地方，也加深了对整个系统的理解。
